# AI Agents from Scratch --- Tool Calling, Agent Loops, and Dynamic Tool Creation

A practical learning repository for understanding how AI agents actually
work at the code and architecture level.

This repository builds three projects step by step:

1.  **Single-Tool Decision Agent** --- the LLM decides whether a
    developer-provided tool is needed.
2.  **Multi-Step Agent Loop** --- the LLM repeatedly chooses actions,
    observes tool results, and continues until the goal is complete.
3.  **Self-Extending Agent** --- the agent detects a missing capability,
    requests code generation, validates and tests the generated tool,
    registers it dynamically, and then uses it.

The goal is not to hide agent behavior behind frameworks. The examples
focus on the mechanics: what is sent to the LLM API, what the model
actually sees, who executes tools, how observations return to the model,
where the loop lives, and how dynamic tool creation can be designed.

> **Core idea:** An agent does not need to write its own tools. A system
> becomes agentic when the model can choose actions based on a goal and
> observations. Dynamic tool generation is a more advanced capability
> layered on top.

## How to Use This Repository

Run the projects in order. Project 1 teaches the model/application
boundary. Project 2 adds the repeated action-observation loop. Project 3
adds a controlled mechanism for creating a missing analytical
capability.

``` bash
python project_1/app.py
python project_2/app.py
python project_3/app.py
```

Watch the terminal output carefully. The most useful things to inspect
are:

-   the tool selected by the model,
-   the arguments generated by the model,
-   the Python function executed by the application,
-   the observation returned to the model,
-   the next action chosen after that observation,
-   and the condition that ends the loop.

## Mental Model

``` text
LLM Call:
User → LLM → Answer

Workflow:
Developer-defined Step A → Step B → Step C

Tool-Using Agent:
User Goal → LLM Decision → Tool → Observation → LLM → Answer

Multi-Step Agent:
Goal → Decide → Act → Observe → Decide → Act → Observe → Finish

Self-Extending Agent:
Goal → Detect Missing Capability → Generate Tool → Validate → Test → Register → Use → Finish
```

The Python application remains responsible for execution. The LLM
selects actions; it does not directly execute your local Python
functions merely because their schemas were included in an API request.

# Project 1 --- Single-Tool Decision Agent

## Goal

Build the simplest tool-using agent.

The user asks:

> What is my name?

The application has one developer-written function:

`get_customer_profile(customer_id)`

The LLM sees the tool name, description, and input schema. It decides
whether the tool is required.

## Architecture

``` text
USER
 │
 │ "What is my name?"
 ▼
LLM
 │
 │ Reads available tool descriptions
 │ Decides profile information is required
 ▼
get_customer_profile(customer_id)
 │
 ▼
{
  "name": "Revanth",
  "city": "Bengaluru"
}
 │
 │ Observation returned to LLM
 ▼
LLM
 │
 ▼
"Your name is Revanth."
```

If the user asks something unrelated, such as:

> Write a poem about the moon.

the model should answer directly without calling the customer-profile
tool.

``` text
                 USER
                   │
                   ▼
                  LLM
                /     \
               /       \
        Tool needed?   No tool needed
             │               │
             ▼               ▼
          Call tool       Answer directly
             │
             ▼
         Observation
             │
             ▼
            LLM
             │
             ▼
           Answer
```

## Important Implementation Detail

You do **not** pass the Python function implementation itself to the
model.

The LLM receives a schema conceptually like:

``` text
Tool name:
get_customer_profile

Description:
Retrieve the current customer's profile information.

Input:
customer_id: string
```

The LLM may request:

``` json
{
  "name": "get_customer_profile",
  "arguments": {
    "customer_id": "CUST001"
  }
}
```

Your application executes:

``` python
get_customer_profile(customer_id="CUST001")
```

The boundary is:

``` text
LLM → DECIDES
APPLICATION → EXECUTES
```

# Project 2 --- Multi-Step Agent Loop

## Goal

Build an agent where one tool call is not enough.

The user asks:

> I have ₹20,000. Can I afford a 3-day trip from Bengaluru to Goa?

Available tools:

-   `get_hotel_cost`
-   `get_travel_cost`
-   `get_food_cost`
-   `calculate_total_cost`

The LLM decides what information is needed, calls tools, observes
results, and continues until enough information exists to answer the
user.

## Architecture

``` text
USER
 │
 │ "Can I do a 3-day Goa trip for ₹20,000?"
 ▼
AGENT
 │
 │ Need hotel cost
 ▼
get_hotel_cost()
 │
 ▼
Observation: hotel cost
 │
 ▼
AGENT
 │
 │ Need travel cost
 ▼
get_travel_cost()
 │
 ▼
Observation: travel cost
 │
 ▼
AGENT
 │
 │ Need food cost
 ▼
get_food_cost()
 │
 ▼
Observation: food cost
 │
 ▼
AGENT
 │
 │ Need total
 ▼
calculate_total_cost()
 │
 ▼
Observation: total trip cost
 │
 ▼
AGENT
 │
 │ Goal complete?
 ├── NO → choose another action
 │
 └── YES
       │
       ▼
FINAL ANSWER
```

The generic loop is:

``` text
              ┌─────────────────────┐
              │                     │
              ▼                     │
USER ──────► LLM                    │
              │                     │
              │ choose action       │
              ▼                     │
            TOOL                    │
              │                     │
              ▼                     │
         OBSERVATION ───────────────┘
              │
              │ goal complete
              ▼
         FINAL ANSWER
```

## Why This Is More Agentic

Project 1 is approximately:

``` text
Question
  ↓
Tool?
  ↓
Answer
```

Project 2 is:

``` text
Goal
 ↓
Choose next action
 ↓
Act
 ↓
Observe
 ↓
Choose next action
 ↓
Act
 ↓
Observe
 ↓
...
 ↓
Finish
```

The agentic decision becomes:

> What should I do next based on the goal and everything observed so
> far?

## What Is Happening in the Code?

1.  The user goal is sent to the LLM.
2.  Tool schemas are sent with the request.
3.  The LLM chooses one or more tools.
4.  Python executes the requested functions.
5.  Results are converted into tool observations.
6.  Observations are sent back to the LLM.
7.  The LLM decides the next action.
8.  The loop continues until the model stops requesting tools and
    returns a final answer.
9.  `MAX_ITERATIONS` prevents an uncontrolled infinite loop.

The core loop is:

``` python
for iteration in range(MAX_ITERATIONS):
```

The finish condition is:

``` python
if not tool_outputs:
    print(response.output_text)
    break
```

# Project 3 --- Self-Extending Agent with Dynamic Tool Generation

## Repository Description

A self-extending AI agent built with Python, Grok API, and OpenAI SDK.
The agent autonomously selects tools, identifies missing capabilities,
generates new Python tools at runtime, validates and tests them in a
controlled environment, dynamically registers successful tools, and uses
them to complete user goals.

## Goal

Build an agent that can detect when its existing tools are insufficient.

Example user goal:

> Calculate the revenue contribution percentage of every product in the
> sales dataset.

Initial dataset:

``` text
product     price     quantity
Laptop      50000     2
Mouse       1000      20
Keyboard    3000      10
Monitor     15000     5
```

Initial tools:

-   `list_columns`
-   `describe_data`
-   `request_new_tool`

There is initially no tool that calculates revenue contribution.

## Architecture

``` text
USER GOAL
   │
   ▼
MAIN AGENT
   │
   │ What capabilities are available?
   ▼
TOOL REGISTRY
   │
   ├── list_columns
   ├── describe_data
   └── request_new_tool
   │
   ▼
MAIN AGENT
   │
   │ Existing analytical tools cannot perform
   │ the required calculation
   ▼
REQUEST TOOL CREATION
   │
   ▼
TOOL BUILDER LLM
   │
   │ Generate Python function
   ▼
GENERATED CODE
   │
   ▼
AST VALIDATION
   │
   ▼
TEST EXECUTION
   │
   ├── FAIL → reject or repair
   │
   └── PASS
         │
         ▼
    REGISTER TOOL
         │
         ▼
      MAIN AGENT
         │
         ▼
      CALL NEW TOOL
         │
         ▼
       RESULT
         │
         ▼
    FINAL ANSWER
```

## The Key Idea: Meta-Tools

A normal agent receives domain tools such as:

``` text
read_csv
list_columns
describe_data
```

A self-extending system also receives a meta-tool such as:

``` text
request_new_tool
```

Conceptually:

``` text
                 AGENT
                   │
         ┌─────────┼─────────┐
         │         │         │
         ▼         ▼         ▼
      Existing   Reasoning   Meta-tools
       tools                 │
                             ├── generate code
                             ├── validate code
                             ├── test code
                             └── register tool
```

The LLM does not magically gain arbitrary capabilities. Developers still
define the foundational environment and permissions.

The developer must control:

-   File access.
-   Network access.
-   API credentials.
-   Database permissions.
-   Shell access.
-   Package installation.
-   CPU and memory limits.
-   Timeouts.

## Main Agent vs Tool Builder

Two logical roles are useful:

``` text
              MAIN AGENT
                  │
                  │ Missing capability
                  ▼
              TOOL BUILDER
                  │
                  ▼
             Generated Code
                  │
                  ▼
                TESTER
                  │
            ┌─────┴─────┐
            │           │
          Failed       Passed
            │           │
            ▼           ▼
         Repair      Register
                        │
                        ▼
                    Main Agent
```

These roles can use the same underlying model. They do not require two
different LLM providers.

The Main Agent focuses on:

-   What is the goal?
-   What information is available?
-   What action should happen next?
-   Is a capability missing?
-   Is the task complete?

The Tool Builder focuses on:

-   What function is required?
-   What input should it accept?
-   What output should it return?
-   How should the logic be implemented under restrictions?

## Tool Registry Code

``` text
Available in git repo under project_3 folder as tool_registry.py
```

``` python
class ToolRegistry:

    def __init__(self):
        self.tools = {}

    def register(
        self,
        name,
        description,
        parameters,
        function
    ):

        self.tools[name] = {
            "name": name,
            "description": description,
            "parameters": parameters,
            "function": function,
        }

    def list_tools(self):

        result = []

        for name, tool in self.tools.items():

            result.append(
                {
                    "name": tool["name"],
                    "description": tool["description"],
                    "parameters": tool["parameters"],
                }
            )

        return result

    def get_openai_tool_schemas(self):

        schemas = []

        for name, tool in self.tools.items():

            schemas.append(
                {
                    "type": "function",
                    "name": tool["name"],
                    "description": tool["description"],
                    "parameters": tool["parameters"],
                }
            )

        return schemas

    def execute(self, name, arguments):

        if name not in self.tools:
            raise ValueError(f"Tool not found: {name}")

        function = self.tools[name]["function"]

        return function(**arguments)
```

## Tool Creation Flow

The main agent can call:

``` text
request_new_tool(
    requirement="Calculate revenue contribution percentage for each product."
)
```

Behind that function, the application:

1.  Sends the requirement to a Tool Builder LLM.
2.  Requests exactly one restricted Python function.
3.  Parses the generated source code.
4.  Applies AST checks.
5.  Executes tests in a restricted test process.
6.  Rejects failed tools.
7.  Converts passing code into a callable function.
8.  Registers it in the tool registry.
9.  Refreshes tool schemas sent to the main agent.
10. Allows the main agent to call the new tool.

## Example Generated Tool

``` python
def generated_tool(records):

    revenues = []
    total_revenue = 0

    for row in records:

        revenue = row["price"] * row["quantity"]

        revenues.append(
            {
                "product": row["product"],
                "revenue": revenue
            }
        )

        total_revenue += revenue

    result = []

    for item in revenues:

        contribution = (
            item["revenue"]
            / total_revenue
            * 100
        )

        result.append(
            {
                "product": item["product"],
                "revenue": item["revenue"],
                "contribution_percentage": contribution
            }
        )

    return result
```

Before tool creation:

``` text
Tool Registry

```

After successful generation and testing:

``` text
Tool Registry
├── list_columns
```

The main agent loop refreshes available schemas:

``` python
response = client.responses.create(
    model=MODEL,
    previous_response_id=response.id,
    input=tool_outputs,
    tools=get_current_tools()

```

This refresh matters because the newly registered tool must become
visible to the model in the next decision step.

## Safety Note

A Python subprocess with a timeout is useful for a learning prototype,
but it is **not a production-grade security sandbox**.

For production, generated code should run in a genuinely isolated
environment with controls such as:

-   Disposable container or VM.
-   Network disabled by default.
-   Read-only or isolated filesystem.
-   Strict CPU limits.
-   Strict memory limits.
-   Execution timeout.
-   Restricted environment variables.
-   No production secrets.
-   Explicit package allowlist.
-   Input/output validation.
-   Audit logging.
-   Human approval for high-impact actions.

# Final Takeaway

The progression is:

``` text
LEVEL 0 — LLM CALL

User → LLM → Answer
```

``` text
LEVEL 1 — TOOL-USING AGENT

User → LLM → Tool → Observation → LLM → Answer
```

``` text
LEVEL 2 — MULTI-STEP AGENT

           ┌──────────────────────┐
           │                      │
User → Agent → Action → Observation
           │                      │
           └──────────────────────┘
                    │
                  Finish
```

``` text
LEVEL 3 — SELF-EXTENDING AGENT

Goal
 ↓
Agent
 ↓
Identify missing capability
 ↓
Generate tool
 ↓
Validate and test
 ↓
Register tool
 ↓
Use tool
 ↓
Observe result
 ↓
Continue or finish
```

The central architectural distinction is:

> A developer can create the available capabilities while the LLM
> decides how and when to use them. That is already agentic behavior.
> Tool generation adds a further layer in which the agent can request a
> new capability, but the environment, validation process, permissions,
> and execution boundaries still need to be designed by the developer.

## Learning Objectives

After completing the three projects, you should be able to explain:

-   the difference between an LLM call, workflow, and agent;
-   why tool descriptions and JSON schemas matter;
-   why the Python function itself is not sent to the LLM;
-   how a model requests a function call;
-   how application code routes and executes that call;
-   how tool outputs become observations;
-   how an agent loop decides whether to continue or finish;
-   why iteration limits are necessary;
-   how a dynamic tool registry works;
-   how a meta-tool can request a missing capability;
-   and why generated code requires strong isolation before production
    use.

## Scope

These projects are intentionally educational. They prioritize
architectural clarity over production complexity. Real systems should
add authentication, structured logging, tracing, retries, schema
validation, model-output validation, permission boundaries,
observability, evaluation datasets, and proper sandboxing for generated
code.
